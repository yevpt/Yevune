# macOS 收藏与评分闭环设计

- **日期**：2026-07-18
- **范围**：OpenSubsonic 标注读取、Rust core 写操作、macOS 原生交互
- **状态**：已实施

## 1. 目标

Yevune 服务端已经实现标准 OpenSubsonic `star`、`unstar` 与 `setRating`，但媒体浏览响应没有带回当前用户的标注，Rust core 和 macOS 客户端也没有操作入口。结果是用户无法在播放或整理时收藏歌曲，也无法看到自己的既有收藏与评分。

本切片补齐端到端闭环：当前用户在曲库、搜索、专辑和当前播放页看到一致的收藏/评分状态，并能通过标准端点修改；失败时保留原状态并给出局部错误，不干扰播放。

## 2. 协议与数据

继续使用 OpenSubsonic，不新增 `/rest/ext/*`：

- `star?id=` / `unstar?id=`：曲目。
- `star?albumId=` / `unstar?albumId=`：专辑。
- `star?artistId=` / `unstar?artistId=`：艺人。
- `setRating?id=&rating=0...5`：按不透明 ID 类型设置或清除评分。

`Artist`、`Album`、`Track` 契约增加两个可空字段：

- `starred: String?`：OpenSubsonic 收藏时间；非空即已收藏。
- `userRating: UInt8?`：当前用户评分 1–5。

字段可空并保持向后兼容。服务端从本地 SQLite `annotations` 按已认证用户批量读取，只附加该用户自己的状态；不得跨用户泄漏。

## 3. 服务端与 core 边界

- 服务端在 `getArtists`、`getArtist`、`getAlbum`、`getSong`、`getAlbumList2`、`getIndexes`、`search3` 与 `getPlaylist` 的最终可见结果上批量附加标注。
- 批量查询有界，跟随每个端点既有最大结果数；禁止逐条 N+1 查询。
- 访问控制仍先过滤媒体，再附加标注；标注不能使不可见媒体重新出现。
- Rust core 负责协议请求和 UniFFI DTO，不在 Swift 中拼 URL 或认证参数。
- 写操作成功后重新读取权威媒体对象，并在当前登录会话内维护轻量标注快照，使播放页与曲库入口保持一致；不做持久化缓存，也不缓存完整媒体对象。

## 4. macOS 交互

- 当前播放页歌曲身份区提供醒目的收藏按钮；点击后乐观锁定控件，成功更新当前曲目的权威快照，失败保持原图标并显示局部错误。
- 搜索曲目、专辑曲目与歌单曲目的上下文菜单提供“收藏/取消收藏”和 1–5 星评分；菜单状态来自 DTO。
- 专辑详情头部提供专辑收藏；艺人详情头部提供艺人收藏。
- 收藏与评分是个人状态，普通用户与管理员使用同一套交互。
- 不把收藏列表塞进播放专注页；收藏专辑继续通过曲库筛选访问。

## 5. 状态与并发

- 同一标注操作 single-flight；操作中禁用对应按钮。
- 成功后重新读取目标实体，避免客户端自行伪造服务端时间戳。
- 切歌或切换详情期间，旧请求结果不得覆盖新目标。
- 失败只显示安全错误文案，不暴露认证 URL。

## 6. 测试与验收

- contract：字段 JSON 名称、缺失字段兼容。
- server：三种实体的读取、搜索/列表返回、用户隔离、批量查询。
- core：解码标注字段；三类收藏参数与评分端点准确；错误信封传播。
- macOS：状态机 single-flight、成功刷新、失败保留、切换目标防陈旧写入；静态呈现测试覆盖各入口。
