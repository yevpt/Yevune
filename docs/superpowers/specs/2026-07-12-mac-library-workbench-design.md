# Mac 曲库管理工作台设计

## 1. 目标

解决 M1 客户端操作状态割裂的问题：上传必须有逐文件成功反馈，扫描必须说明新增/更新/删除了哪些内容，完成后曲库列表必须自动刷新。主界面采用已确认的 A1 方案：单窗口工作台、全窗口拖放、工具栏入口与底部任务抽屉。

## 2. 范围

- 主窗口内完成文件选择/拖放、批量上传、自动扫描、手动扫描、结果查看和曲库刷新。
- 服务端在 `/rest/ext/*` 提供详细扫描报告；标准 OpenSubsonic 扫描端点行为不变。
- 不实现后台常驻任务、跨启动任务历史、通知中心、离线下载或管理员/访问控制 UI。

## 3. 界面结构

主窗口继续使用 `NavigationSplitView`：左侧为专辑列表，右侧为搜索或专辑详情。工具栏提供“导入音乐”“扫描曲库”和任务抽屉开关。

窗口整体接受多个本地音频文件拖放。拖入时显示覆盖层；接收后底部任务抽屉自动展开。抽屉逐项显示文件名、进度以及等待、上传中、成功、失败状态。成功必须使用明确文字和图标，不能只留下进度条。

扫描任务先显示汇总：新增、更新、删除、未变化数量；展开后按动作列出曲目标题、艺人、专辑。完成后自动刷新左侧专辑，新增专辑短暂显示“新增”标记，并提供“查看新增内容”操作。

## 4. 工作流与状态

用户一次拖放或选择的文件形成一个导入批次。Swift 工作流 ViewModel 依次调用 core 上传接口，并保留每个上传返回的 `Track`。本批所有可上传文件结束后，对共同的 `library/` 前缀触发一次详细扫描；个别上传失败不阻止其他文件继续，但任务区分别显示失败原因。

自动扫描成功后立即刷新曲库。扫描失败时保留已经成功的上传状态，显示“文件已上传，但索引失败”和“重试扫描”。手动扫描使用同一详细扫描接口，完成后执行同样的刷新逻辑。

## 5. 服务端与 core

扩展端点 `/rest/ext/startScan` 保持管理员权限和前缀参数，响应 `scanResult` 增加 `changes` 数组。每项包含动作 `added|updated|deleted`、对象键，以及可获取时的曲目 DTO；删除项至少包含删除前保存的曲目 DTO，确保 UI 能说明被删除内容。

扫描器的内部报告增加有界变更集合。单次响应最多返回 500 项明细，并同时返回 `changesTruncated`，防止超大扫描产生无界内存；计数始终是完整计数。扫描过程仍按对象流式读取，不改变资源红线。

core 新增 UniFFI 记录 `DetailedScanResult` 与 `ScanChange`，并暴露 `scan_prefix(prefix)`。上传门面继续从本地路径流式传输，但 Swift 协议返回服务端生成的 `Track`，不再丢弃结果。Swift 不解析 HTTP、不复制扫描规则。

## 6. 错误处理

- 单文件上传失败只标记该项，批次继续。
- 扫描已在运行、认证失败或服务端失败显示可重试错误。
- 曲库刷新失败不改写已完成的上传/扫描结果，单独显示“刷新失败”。
- 非音频或无法读取的拖放项在发起请求前显示本地失败状态。

## 7. 测试

- 服务端 TDD：详细报告覆盖新增、更新、删除、截断和管理员权限。
- core TDD：解码详细扫描响应、前缀参数及上传返回 Track。
- XCTest：逐文件成功/失败反馈、批次完成后自动扫描、扫描失败保留上传成功、扫描后刷新专辑和新增标记。
- 端到端：真实 Docker 服务完成拖放上传 → 自动扫描 → 明细显示 → 左侧专辑刷新。
- 全量执行 Rust test/clippy/fmt、Swift test 与一键启动脚本测试。

## 8. 架构边界

网络、认证、上传和扫描 DTO 处理位于 Rust core；Swift ViewModel 只协调 UI 状态，SwiftUI 只渲染和转发用户操作。音频保持本地路径跨 FFI 与流式有界传输，客户端不持 Garage 凭证。
