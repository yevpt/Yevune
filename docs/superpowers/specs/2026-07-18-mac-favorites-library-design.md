# macOS 统一收藏曲库设计

- **日期**：2026-07-18
- **范围**：OpenSubsonic `getStarred2`、Rust core、macOS 原生收藏入口
- **状态**：已实施

## 1. 目标

收藏与评分写操作已经贯通，但当前“我的收藏”只藏在专辑筛选中，收藏歌曲和艺人没有集中入口。用户需要记住媒体所在位置才能再次找到它，不符合主流音乐客户端的收藏体验。

本切片增加侧栏一级“收藏”入口，在一个页面集中浏览当前用户收藏的歌曲、专辑和艺人。歌曲可以直接播放并继续使用现有上下文整理动作；专辑和艺人可进入既有详情。当前播放页保持专注，不增加收藏列表或“接下来播放”。

## 2. 协议与边界

- 使用标准 OpenSubsonic `getStarred2`，响应键为 `starred2.artist`、`starred2.album`、`starred2.song`。
- 服务端按 `starred_at DESC, id DESC` 返回三类媒体，只读取当前认证用户的标注。
- 曲库访问控制在 SQL 查询阶段强制；不可见媒体不得因收藏重新出现。
- DTO 继续复用 `contract::Artist`、`Album`、`Track`，服务端补齐 `starred` 与 `userRating`。
- Rust core 独占协议、认证与响应解码；Swift 不拼接 URL。
- 不新增依赖，不引入扩展端点。

## 3. macOS 信息架构

- 侧栏“资料库”分组包含“曲库”和“收藏”，收藏使用 `heart` 图标。
- 收藏页顶部展示标题、三类数量和刷新按钮；主体用“歌曲 / 专辑 / 艺人”分段切换，默认歌曲。
- 歌曲采用紧凑列表，Return/双击从所选歌曲开始播放整份收藏歌曲队列；上下文菜单复用播放、加入歌单、收藏与评分动作。
- 专辑与艺人使用原生自适应网格；点击进入既有专辑/艺人详情，不复制详情实现。
- 取消收藏成功后从当前收藏列表移除；其他入口产生的标注变更在重新进入或手动刷新时读取权威结果。
- 加载失败保留已有内容并显示局部错误；首次失败提供重试，不用空白页替代。

## 4. 状态与并发

- `FavoriteLibraryViewModel` 每次加载使用 generation 隔离陈旧响应，并复用同一进行中任务。
- 收藏列表保持服务端顺序；客户端不按标题重新排序。
- 页面内取消收藏通过会话级 `MediaAnnotationViewModel` 执行，成功后由列表模型按目标移除，失败保留条目。
- 离开收藏页不影响 `PlaybackController`，切换媒体详情也不重建播放会话。

## 5. 测试与验收

- server：三类收藏、时间顺序、用户隔离、访问控制、XML/JSON 响应键。
- core：`getStarred2` 请求与三类字段解码、错误信封传播。
- macOS：加载/刷新/陈旧响应、取消收藏移除、播放队列起点、侧栏与静态交互入口。
- 真实窗口验收覆盖歌曲播放、分段切换、专辑/艺人详情与取消收藏即时移除；同时修复艺人详情首次呈现为空时不会触发加载的问题。
- 完成前运行 Swift 全量测试、Rust 全量测试、Clippy、fmt、OpenAPI 漂移检查，并在真实 macOS 窗口验证紧凑与常规布局。
