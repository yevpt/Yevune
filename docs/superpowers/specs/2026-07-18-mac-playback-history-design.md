# macOS 播放历史闭环设计

- **日期**：2026-07-18
- **范围**：OpenSubsonic `scrobble`、Rust core、macOS 全局播放器
- **状态**：已实施

## 1. 目标

macOS 客户端已经提供“最近播放”和“最常播放”专辑筛选，但全局播放器没有上报播放行为，导致两类曲库视图无法反映本客户端的真实使用。本切片让播放行为通过标准 OpenSubsonic `scrobble` 回写当前用户历史，同时保持播放主链路不受上报网络请求影响。

## 2. 协议与边界

- Rust core 新增 `scrobble(trackId, submission)`，统一负责认证、参数编码和错误信封处理；Swift 不拼接请求 URL。
- 媒体成功装载后发送 `submission=false` 的开始播放通知。
- 有效播放达到阈值后发送一次 `submission=true`；自然播放结束也兜底提交一次。
- 上报失败是非阻断故障：不暂停、不跳曲、不覆盖播放错误状态，也不在日志中泄露认证 URL。
- 不新增依赖，不改服务端协议或数据库结构。

## 3. 有效播放判定

- 每次媒体装载形成独立播放实例；同一曲目重复播放可以分别计数。
- 阈值为曲目时长的 50%，最长 4 分钟；短于 60 秒的曲目仅在自然结束时计数，避免预览式点播污染历史。
- 只累计播放器处于 `playing` 时相邻进度事件的正常前进量；暂停时间、倒退和大跨度跳转不计入实际聆听时间。
- 单个相邻增量最多计 5 秒，防止拖动时间轴直接越过阈值后立即上报。
- 同一播放实例的完成提交最多发送一次；媒体重试不创建新的播放实例，真正切换或单曲循环重新装载才重置计数。

## 4. 并发与生命周期

- 上报任务与媒体解析、装载和队列推进解耦，不等待网络完成才开始或切换歌曲；请求按产生顺序在后台串行发出，完成提交不会越过开始通知。
- generation 隔离陈旧上报结果；陈旧任务不能修改当前播放实例的计数状态。
- 登出或播放器关闭时取消尚未发出的上报任务；服务端已经收到的请求不做补偿回滚。
- Swift 测试使用协议化 reporter，不依赖网络；生产 reporter 复用同一个 `CoreMusicClient` 会话。

## 5. 测试与验收

- core：开始与完成上报使用 `/rest/scrobble`、正确携带 `id` 和 `submission`，错误信封正常传播。
- macOS：装载后开始上报、正常聆听达到阈值仅提交一次、跳转不误计数、自然结束兜底、切歌隔离、上报失败不影响播放。
- 完成前运行 Swift、core 与受影响服务端测试，以及 Clippy、fmt 和 `git diff --check`。
- 已完成 Swift 全量测试与 release 构建、Rust 全量测试、Clippy、fmt、服务端播放历史回归和启动脚本回归。
