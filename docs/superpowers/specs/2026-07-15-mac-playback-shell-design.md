# macOS 现代播放壳层设计

- **日期**：2026-07-15
- **子项目**：Yevune macOS 原生客户端
- **里程碑**：M3 — 在线播放壳层
- **状态**：设计已确认

## 1. 背景与目标

Yevune 的 macOS 客户端已经具备曲库浏览、专辑与歌单管理、标签整理、用户与角色管理及访问控制，但播放能力仍是专辑详情内部的一次性“试听”：播放器绑定在 `MediaViewModel`，离开局部视图后没有可见的全局播放状态，也没有队列、时间轴、音量、系统媒体键、专注播放页或迷你播放器。

M3 要把这套局部试听升级为应用级原生播放器，使客户端既能管理曲库，也能承担日常播放。成功标准如下：

1. 从专辑、歌单和搜索结果发起播放时，使用同一个全局播放器和队列。
2. 页面切换和管理操作不打断播放。
3. 队列非空时显示常驻底部播放栏，提供主流即时控制。
4. 提供专注当前歌曲的沉浸式播放页，不混入队列或曲库管理信息。
5. 支持系统媒体键、系统播放信息和独立迷你播放器。
6. 网络或媒体 URL 失败时可恢复、可解释，不产生无限重试。

本阶段只覆盖在线流媒体播放。离线下载与歌词获取/逐行同步分别作为后续切片；M3 为它们保留稳定接入点，但不以占位实现冒充完整能力。

## 2. 架构边界

沿用现有客户端架构，不新增后端语言、数据库、服务或第三方播放依赖：

```text
contract Track DTO
       │
       ▼
core MusicClient ── 生成认证 stream URL / cover URL
       │ UniFFI
       ▼
PlaybackController（应用级状态与编排）
       ├── PlaybackQueue（纯队列状态）
       ├── PlaybackEngine（AVQueuePlayer 适配）
       ├── SystemMediaCoordinator（媒体键与系统播放信息）
       └── SwiftUI 视图（底栏 / 专注页 / 队列面板 / 迷你播放器）
```

- `core` 继续负责认证、协议和媒体 URL 生成，不负责出声或 macOS 播放生命周期。
- Apple 客户端使用 AVFoundation 播放，符合既有 ADR-0001 与 Mac 客户端设计。
- 播放队列属于运行期平台体验，M3 先在 Swift 侧实现；不为尚未出现的跨端队列同步扩大 UniFFI 接口。
- 视图不直接持有 `AVPlayer`，也不各自创建播放器。

选择全局 `PlaybackController + AVQueuePlayer`，而不是单个 `AVPlayer` 手写换曲，原因是系统队列能力、预装载和切换事件更贴近需求；也不把播放状态搬进 Rust `core`，避免跨 FFI 同步平台播放器生命周期。

## 3. 组件职责

### 3.1 `PlaybackQueue`

纯 Swift 状态模型，不依赖 AVFoundation：

- 保存原始播放上下文、当前曲目位置和待播放顺序。
- 保留歌单中的重复曲目，因此队列项使用独立实例标识，不能只按 `track.id` 去重。
- 实现立即播放、下一首播放、追加、删除、移动、清空、上一首和下一首。
- 实现随机播放、单曲循环和列表循环。
- 开启随机时保留当前曲目，只重排尚未播放的队列项；关闭时恢复原上下文中剩余曲目的相对顺序。
- 专辑上下文按碟号、曲目号排序；歌单上下文严格保留服务端返回顺序。

### 3.2 `PlaybackEngine`

对 AVFoundation 的薄适配层：

- 以协议暴露装载、播放、暂停、跳转、进度、音量和静音能力。
- 生产实现内部使用 `AVQueuePlayer`，把时间、缓冲、结束和失败事件转换成可测试的播放器事件。
- 集中管理周期观察者与通知观察者，替换媒体或释放时必须清理，避免重复回调和生命周期泄漏。
- 不包含业务队列规则、URL 重试策略或界面文案。

### 3.3 `PlaybackController`

应用级 `@MainActor ObservableObject`，是播放状态的唯一真相源：

- 协调 `PlaybackQueue`、`PlaybackEngine` 和媒体 URL 解析。
- 发布当前曲目、播放/缓冲/失败状态、经过时间、时长、音量、循环模式和队列快照。
- 将队列动作转成播放器装载与切换动作。
- 处理认证 URL 刷新、单次重试、坏曲跳过和登出清理。
- 向系统媒体协调器推送当前曲目和播放进度。
- 提供视图所需的意图方法，视图不拼装业务状态变化。

控制器由登录后的应用壳层创建一次，并注入曲库、专辑、歌单、搜索、底栏、专注页和迷你播放器。登出时销毁播放会话并清除带认证信息的资源。

### 3.4 `SystemMediaCoordinator`

- 使用 `MPRemoteCommandCenter` 接收播放、暂停、上一首、下一首和跳转命令。
- 使用 `MPNowPlayingInfoCenter` 发布曲名、艺人、专辑、封面、时长、进度和播放速率。
- 系统命令映射到 `PlaybackController` 的同一意图方法，不建立第二套状态。
- 控制器注销或登出时移除命令目标并清空系统播放信息。

### 3.5 SwiftUI 视图

- `PlayerBar`：队列非空时常驻主窗口底部。
- `NowPlayingView`：专注当前歌曲的沉浸式播放页。
- `QueuePanel`：从曲库底栏或应用菜单单独打开的队列管理面板。
- `MiniPlayerView`：独立 macOS 窗口，复用全局控制器。

视图按职责拆成小文件，避免继续放大现有 `LibraryView.swift` 和 `MediaViewModel.swift`。

## 4. 播放行为

### 4.1 发起播放

- 双击曲目：以当前专辑、歌单或搜索结果为上下文，从所点曲目开始播放。
- “立即播放”：用所选曲目或上下文替换当前队列。
- “下一首播放”：插入当前曲目之后。
- “加入队列”：追加到队尾。
- 播放入口使用一致的菜单名称与图标，不继续显示“试听”。

### 4.2 导航与时间轴

- 点击上一首始终直接切换到队列中的上一首，不根据已播放时长重播当前曲目。
- 下一首直接推进；曲目自然结束时使用相同推进规则。
- 时间轴支持点击和拖动跳转，并在拖动期间显示预期时间，提交后再同步播放器。
- 缓冲和用户主动暂停是两个独立状态，界面不得把网络等待显示成“已暂停”。

### 4.3 随机与循环

- 默认顺序播放。
- 随机播放只改变尚未播放部分；当前曲目不跳变。
- 循环模式按“关闭 → 列表循环 → 单曲循环”切换。
- 单曲循环在当前媒体结束时从头播放，不改动队列位置。

## 5. 界面与交互

### 5.1 常驻底部播放栏

队列非空时，主窗口底部播放栏始终可见：

- 左侧：封面、曲名、艺人；点击封面或歌曲信息进入专注播放页。
- 中间：上一首、播放/暂停、下一首和可拖动时间轴；显示已播放时间与总时长。
- 右侧：随机、循环、音量与静音；队列入口位于曲库工作状态的底栏或应用菜单。
- 播放栏不能遮挡任务抽屉；两者由应用壳层统一安排安全区和层级。

### 5.2 专注播放页

专注页只呈现当前歌曲，不显示“接下来播放”、队列或曲库管理信息：

- 左侧以大封面、曲名、艺人和专辑建立歌曲身份。
- 中央区域作为滚动歌词的唯一动态主舞台。
- 底部只保留上一首、播放/暂停、下一首、时间轴、随机、循环、音量和静音。
- 左上角明确返回曲库；返回不会停止播放。
- 无歌词时中央区域显示安静、明确的无歌词状态，不生成假歌词或占位行。

歌词列是稳定布局约束。后续歌词切片接入真实数据时，不得把队列重新放进专注页，也不得用浮层覆盖歌词。

### 5.3 队列面板

队列是独立工具面板，只从曲库底栏或应用菜单打开：

- 标记当前曲目，并展示后续队列。
- 支持拖动排序、删除单项和清空待播放项。
- 操作后立即同步 `PlaybackQueue` 与 `AVQueuePlayer`。
- 关闭面板不影响播放；专注页默认不提供该面板的视觉入口。

### 5.4 迷你播放器

- 独立窗口显示封面、曲名、艺人、上一首、播放/暂停、下一首和简化时间轴。
- 使用同一 `PlaybackController`，打开或关闭窗口不创建、替换或暂停播放器。
- 窗口支持置顶由用户控制；首版不引入自定义无边框窗口行为。

### 5.5 视觉方向

- 保持原生 macOS 窗口、材质、键盘焦点和可访问性，不仿制移动端底部导航。
- 日常曲库界面保持克制，底栏作为稳定控制基线。
- 专注页把视觉风险集中在“封面 + 大面积歌词舞台”：允许由当前封面色调形成低对比背景氛围，但文字对比度和减少动态效果设置优先。
- 动效只用于页面展开、当前歌词行过渡和播放状态变化；遵守系统 Reduce Motion。

## 6. 数据流

以专辑中双击曲目为例：

1. `MediaDetailView` 将专辑曲目和起始曲目实例交给 `PlaybackController.play(context:startingAt:)`。
2. 控制器创建队列快照，请求当前曲目的 `core.stream_url` 和封面 URL。
3. URL 转换成功后，控制器将媒体交给 `PlaybackEngine` 并开始播放。
4. Engine 事件更新控制器的播放状态、进度和缓冲状态。
5. `PlayerBar`、`NowPlayingView`、`MiniPlayerView` 与系统播放信息观察同一状态。
6. 曲目结束时控制器按队列及循环模式解析下一项，获取新 URL 并推进。

认证 URL 只能短期存在于运行时媒体项中。错误、遥测和调试日志不得输出完整 URL 或认证查询参数。

## 7. 故障恢复与边界状态

### 7.1 URL 与媒体失败

- 首次生成 URL 失败：保留队列与当前位置，显示可理解的错误状态。
- URL 过期或媒体装载失败：重新向 `core` 获取 URL 并自动重试一次。
- 同一队列实例重试仍失败：记录该实例本轮已失败，提示用户并推进到下一首。
- 自动推进不得再次尝试本轮已经失败的实例，避免列表循环产生无限失败环。
- 用户手动再次点击该曲目视为新的播放尝试，可重新解析 URL。

### 7.2 网络波动

- AVFoundation 正在等待数据时显示缓冲状态并保持当前位置。
- 短时断网不清空队列、不重置进度，也不立即跳曲。
- 只有收到明确失败后才进入 URL 刷新与跳过流程。

### 7.3 会话与生命周期

- 页面切换、管理操作、打开或关闭迷你播放器均不停止播放。
- 登出时立即暂停、清空队列、释放媒体项、移除观察者，并清空系统播放信息。
- 首版不跨应用退出恢复队列；应用重新启动后从空队列开始。
- 队列为空时隐藏底栏并关闭专注页/迷你播放器中的可操作播放状态。

## 8. 后续歌词接入点

M3 定义 `LyricsState` 的界面状态边界：`unavailable`、`loading`、`unsynced(text)`、`synced(lines,currentLine)` 与 `failed(message)`。首版仅使用 `unavailable`，不新增歌词协议或服务端端点。

后续歌词切片应优先评估 OpenSubsonic 的歌词能力，再决定是否需要 `/rest/ext/*` 扩展；任何跨端接口必须先更新 `contract`。逐行同步由独立歌词模型消费播放进度，不进入 `PlaybackQueue` 或 `PlaybackEngine`。

## 9. 测试策略

### 9.1 队列单元测试

- 专辑排序、歌单顺序和重复曲目实例。
- 立即播放、下一首播放、追加、移动、删除和清空。
- 直接切换上一首和下一首的边界行为。
- 随机开启/关闭时当前曲目稳定及剩余顺序恢复。
- 无循环、列表循环和单曲循环的结束行为。

### 9.2 控制器单元测试

使用假的 `PlaybackEngine`、媒体 URL 解析器和确定性随机源验证：

- 首曲装载、播放/暂停、进度跳转、音量和静音。
- Engine 事件到公开 UI 状态的映射。
- URL 刷新只重试一次，失败后跳过且不形成无限循环。
- 页面消费者增减不影响同一播放会话。
- 登出完整清理队列、观察者、媒体项和系统信息。

### 9.3 系统集成测试

- 远程播放、暂停、上一首、下一首和跳转命令映射正确。
- 当前曲目、封面、时长、进度和播放速率更新正确。
- 多次注册/注销不产生重复命令处理。

### 9.4 macOS 冒烟测试

在真实服务端与真实音频上验证：

1. 从专辑中间曲目开始，按专辑顺序继续播放。
2. 从含重复曲目的歌单开始并保持顺序。
3. 切换曲库、歌单和管理页，播放不中断。
4. 操作底栏时间轴、音量、随机、循环和独立队列面板。
5. 打开专注页与迷你播放器，三处状态一致且不创建第二个音频会话。
6. 使用键盘媒体键控制并检查系统播放信息。
7. 登出后声音停止，系统播放信息和认证媒体 URL 被清除。

完成前继续执行仓库通用门禁：Swift 测试与构建、`cargo test`、`cargo clippy -- -D warnings`、`cargo fmt --check` 及 macOS 客户端冒烟脚本。

## 10. 实现切片

详细任务在后续实现计划中拆分，设计上按以下垂直切片推进：

1. 纯队列模型与全局控制器骨架。
2. AVQueuePlayer Engine 与真实媒体 URL 播放。
3. 专辑、歌单和搜索播放入口及底部播放栏。
4. 队列面板、随机、循环、进度与音量。
5. 专注播放页。
6. 系统媒体键与系统播放信息。
7. 迷你播放器、故障恢复和完整冒烟验证。

每个切片严格执行失败测试 → 最小实现 → 测试通过 → 小步提交。歌词数据接入与离线下载不混入本计划。
