# Mac 专辑整理台设计

- **日期**：2026-07-17
- **子项目**：Yevune macOS 原生客户端
- **里程碑**：M5 — 专辑整理台
- **状态**：设计已确认

## 1. 背景与目标

M3 已建立应用级播放壳层，M4 已把曲库升级为支持大曲库分页、艺人/专辑浏览、搜索与响应式检视器的发现工作台。当前专辑详情仍沿用早期管理界面：没有明确的加载与刷新状态，快速切换专辑可能被晚到请求覆盖；播放、编辑、移动和删除混在同一级；普通成员仍能看到注定失败的管理入口；批量操作只有失败计数，没有进度、取消和逐项结果；现有标签更新只能“保持或设置”，无法清空已有字段。

M5 将专辑详情升级为适合日常播放与管理员整理的原生 macOS 专辑整理台。成功标准如下：

1. 播放是专辑详情的主操作，管理能力不抢占歌曲身份与播放路径。
2. 普通成员只看到播放、选择和加入个人歌单；管理员才看到修改曲库的入口。
3. 专辑加载、刷新、切换与失败恢复是确定性的，旧请求不得覆盖新专辑。
4. 单曲与批量标签编辑能可靠表达保持、设置和清空，并在提交前校验。
5. 批量写操作显示进度、支持停止后续项目，并列出具体成功、失败与未处理曲目。
6. 更换封面、移动和删除完成后立即刷新真实结果，播放会话全程不中断。
7. 920pt 紧凑详情与宽屏检视器都保持可读、可操作且不溢出。

## 2. 范围与非目标

### 2.1 本阶段包含

- 重构 `MediaDetailView` 的专辑头部、曲目表格、选择与批量操作栏。
- `MediaViewModel` 的初始加载、保留内容刷新、generation、取消和错误分层。
- 管理员专属的标签、封面、移动、删除和可见范围入口守卫。
- 单曲标签编辑器的预填、脏状态、校验、提交状态与显式清空。
- 批量标签编辑的保持/设置/清空三态。
- 批量标签更新与删除的进度、停止、逐项结果和最终刷新。
- 封面替换后的缓存失效与真实封面刷新。
- 原生键盘、VoiceOver、Reduce Motion、空态和失败恢复。

### 2.2 本阶段不包含

- 不改变播放队列、音频引擎、系统媒体信息、专注播放页或迷你播放器。
- 不把“接下来播放”放入专辑详情或专注播放页。
- 不实现歌词获取、滚动歌词、离线下载或跨设备队列同步。
- 不重做导入任务中心；批量导入、取消上传、重试和任务清理作为 M6。
- 不实现音频文件标签写回；M5 继续使用现有数据库覆盖层，不隐式修改 Garage 原文件。
- 不新增批量服务端端点；低频管理员写操作由客户端逐项、有界执行。
- 不新增第三方 UI、状态管理或并发依赖。

## 3. 信息架构与视觉方向

专辑整理台的单一工作是：在不干扰播放的前提下，检查并整理一张专辑。界面从上到下分为专辑头部、曲目表格与按需出现的批量操作栏：

```text
┌──────────────────────────────────────────────────────────┐
│ ┌──────────┐  专辑标题                        [播放全部] │
│ │          │  艺人                                      │
│ │  200 pt  │  年份 · 流派 · 12 首 · 48 分钟       [•••] │
│ │  封面    │                                            │
│ └──────────┘                                            │
├──────────────────────────────────────────────────────────┤
│ 曲序 │ 标题                 │ 艺人       │ 时长 │ 格式  │
│  01  │ Track One            │ Artist     │ 4:21 │ FLAC  │
│  02  │ Track Two            │ Artist     │ 3:58 │ FLAC  │
├──────────────────────────────────────────────────────────┤
│ 已选 3 首        [修改标签] [加入歌单] [更多] [取消选择] │
└──────────────────────────────────────────────────────────┘
```

视觉继续使用系统字体、动态颜色、系统强调色、原生表格焦点与标准材质。封面扩大到约 200pt，是界面唯一强视觉焦点；不增加通用渐变卡片、大面积玻璃层或第三方字体。专辑标题建立主层级，艺人次之，年份、流派、曲目数和总时长压缩为一行可扫描元信息。当前播放曲目只使用扬声器图标与系统强调色，不用整行高饱和铺色。

底部批量栏使用轻量系统材质并固定在详情内容底部，但必须位于全局 `PlayerBar` 之上、内容滚动区域之外。它只在存在选择时出现，并遵守 Reduce Motion；不得遮挡曲目、任务抽屉或播放器。

视觉令牌使用 macOS 语义资源而非固定十六进制颜色，保证系统强调色、深浅模式和提高对比度设置真实生效：

- 窗口底色：`windowBackgroundColor`。
- 表格与轻量表面：`controlBackgroundColor` / 系统 material。
- 主文字：`labelColor`；辅助信息：`secondaryLabelColor`。
- 选中与播放状态：用户的系统 `accentColor`。
- 破坏性动作：`systemRed`，只用于明确的删除入口与确认。

字体角色同样使用原生动态字号：专辑名为 `.largeTitle` 或空间不足时的 `.title`，艺人为 `.title3`，曲目正文为 `.body`，元信息为 `.caption`，时长与曲序使用等宽数字。这里刻意不引入固定品牌色和展示字体，因为 Yevune 的视觉身份来自每位用户自己的唱片封面；固定主题反而会与曲库内容竞争。

本阶段唯一可识别的视觉签名是“唱片标签”：200pt 封面右侧的一行紧凑元信息像实体唱片脊背标签一样把年份、流派、曲目数与总时长串联起来，并与下方高密度曲目表对齐。除这一处外保持安静，不追加装饰性徽章、编号或无业务含义的动效。

## 4. 专辑头部与播放行为

- “播放全部”使用 `PlaybackViewPolicy.albumPlaybackOrder` 生成顺序，并从第一首开始替换当前播放上下文。
- 曲目行播放按钮从该曲目开始播放整张专辑；双击曲目执行相同行为。
- 单击只选择，不自动播放。标准 macOS 多选、Shift 连选、Command 点选和 Command-A 可用。
- 当前曲目显示扬声器状态；暂停与缓冲沿用全局 `PlaybackController` 的既有语义。
- 页面切换、标签编辑、批量操作、封面替换和返回曲库均不得暂停、重建或清空播放会话。
- 专辑管理菜单仅为管理员构造，包含“修改专辑信息”（若由曲目标签实现则明确说明作用范围）、“更换封面”、“可见范围”等非高频入口。
- 移动与删除不放进标签编辑表单；单曲入口位于曲目“更多”菜单，删除始终二次确认。

## 5. 响应式曲目表格

宽屏检视器和紧凑详情使用同一数据模型，但列集合按可用宽度收敛：

- `>= 620pt` 的详情内容：曲序、标题、艺人、时长、格式。
- `< 620pt` 的详情内容：曲序、标题/艺人复合列、时长；格式进入辅助信息或检查器。
- 标题列获得剩余宽度并可截断；固定列不得把内容推出窗口。
- 曲目号由碟号与曲序决定显示；多碟专辑在碟片边界显示安静的分组标签。
- 空专辑显示明确空态；普通成员提示“这张专辑暂时没有可播放曲目”，管理员额外获得导入入口指引，但导入仍由现有全局工作流承担。
- 表格保留可见键盘焦点、选中状态和 VoiceOver 行摘要。上下文菜单与批量栏使用一致动作名称。

紧凑模式返回曲库继续复用 M4 的 `NavigationStack` 路径；详情内部选择在专辑 ID 改变时清空，在同一专辑刷新时保留仍存在的曲目 ID。

## 6. 权限与动作策略

`LibraryBrowserView` 将 `session.admin` 明确传入专辑详情。新增纯策略 `AlbumWorkbenchPolicy`，由输入的管理员状态、宽度、选择数量和操作状态决定可构造的动作与列，不让视图散落权限判断。

普通成员可使用：

- 播放全部、单曲播放和现有播放上下文动作。
- 选择曲目、加入自己的歌单。
- 查看专辑和曲目信息。

管理员额外可使用：

- 修改单曲或批量标签。
- 更换专辑封面。
- 移动和删除曲目。
- 打开专辑、艺人和曲目的可见范围编辑器。

成员模式不得构造以上管理按钮、菜单项、文件选择器或 sheet。客户端守卫只改善体验，服务端 `ApiAdmin` 仍是最终授权边界。M5 同时把现有 `onManageAccess` 接回 `LibraryView` 已持有的 `accessTarget` 路由，且只对管理员传入闭包。

## 7. 确定性加载状态

`MediaViewModel` 发布独立的内容与请求状态：

```swift
enum AlbumDetailPhase: Equatable {
    case idle
    case loading
    case content
    case refreshing
    case failed(String)
}
```

模型保存当前 `albumID`、detail、cover URL、generation 与正在运行的 Task：

- 首次加载无旧内容时显示稳定骨架，不先显示空列表。
- 同一专辑刷新时保留旧内容，使用非阻塞进度与失败横幅。
- 切换专辑立即递增 generation、取消旧 Task，并清除不属于新专辑的内容。
- 每个响应写入前同时核对 generation 与 album ID。即使底层 FFI 不服从取消，晚到结果也不能改变 detail、封面、错误或 phase。
- 详情请求与封面 URL 请求可以并行，但只在同一 generation 内分别发布；封面失败不能把已成功的曲目详情变成整页失败。
- 错误分为初始失败、刷新失败与管理操作失败，后一次加载不得无条件抹掉仍需用户阅读的操作结果。

封面替换成功后重新获取封面 URL，并附加一次仅用于内存/URL 缓存失效的版本标识；不得把认证 URL 写入日志或持久化。真实新封面加载成功后才显示“封面已更新”。

## 8. 标签字段语义与契约扩展

### 8.1 用户语义

单曲编辑器预填当前有效值，并按原值计算字段意图：

- 未改变：保持，不发送该字段。
- 输入新值：设置。
- 将原有可选字段清空：显式清空。

标题是曲目的必填身份，只支持保持或设置，空标题为本地校验错误。专辑、艺人、流派、年份、曲序和碟序支持显式清空。数字字段仅接受各自范围内的正整数；无效文本不静默转换为“保持”。保存按钮在没有变更、存在校验错误或正在提交时禁用。

批量编辑器不能用空文本隐式猜测意图。每个公共字段显式选择：

- 保持：所有选中曲目维持当前值。
- 设置：把输入值应用到所有选中曲目。
- 清空：清空所有选中曲目的该字段。

批量编辑不提供标题、曲序和碟序的统一“设置”为相同值，避免批量制造重复身份；首版批量字段为专辑、艺人、流派和年份，其中年份允许清空。曲序与碟序的批量重排应在未来独立排序工具中设计。

### 8.2 跨端表达

现有 `core::TagUpdate` 的 `Option<T>` 中，`None` 表示保持，无法区分清空。M5 在共享 `contract` 中新增稳定的 `TagField` 枚举，并在 core 的 UniFFI 更新记录中增加 `clear_fields: Vec<TagField>`：

```text
TagUpdate
├── title / album / artist / genre / year / track / discNumber：待设置值
└── clearFields：待显式清空的字段
```

同一字段不能同时设置和清空；core 在发请求前拒绝冲突、空更新、清空 title、未知字段及越界数字。已有 `/rest/ext/updateTags` 保持原参数语义，并兼容增加一个可重复的 `clear` 参数，例如 `clear=genre&clear=year`。这仍是命名空间隔离的自研扩展，不改变任何 OpenSubsonic 标准端点；扩展版本随 `getOpenSubsonicExtensions` 的现有 Yevune 扩展声明递增。

### 8.3 数据库覆盖语义

`tag_overrides.value` 当前已经允许 `NULL`。M5 明确定义：

- 不存在覆盖行：使用扫描自 Garage 文件得到的原标签。
- 覆盖行且 value 非 NULL：使用覆盖值。
- 覆盖行且 value 为 NULL：该字段被显式清空。

因此读取查询不能继续用简单 `COALESCE(override, source)` 判断显式清空，而要先判断覆盖行是否存在。`set_tag_overrides` 接受可空值并在同一事务内 upsert 设置与清空。现有 `clear_tag_overrides` 继续表示“删除覆盖、恢复文件标签”，与“显式清空有效标签”保持不同语义。

所有受覆盖影响的路径必须同步验证：单曲/专辑详情、搜索、排序、流派集合、FTS 同步、访问范围显示和扫描后读取。清空可选字段不能产生字符串 `""`、数字 `0` 或意外回退到文件原值。标题禁止清空，从而保持现有 `Track.title: String` 契约。

## 9. 编辑器交互

标签编辑 sheet 使用标准 macOS 标题栏与工具栏：左侧“取消”，右侧“保存更改”。表单按“歌曲信息”和“排序信息”分组，标签与用户看到的概念一致，不暴露数据库覆盖层或对象键。

- 保存成功后关闭 sheet、刷新当前专辑，并显示与动作同名的结果“标签已更新”。
- 保存失败时保留输入、焦点和脏状态，sheet 内显示具体错误并允许重试。
- 正在提交时禁用重复保存，但允许请求完成后取消关闭。
- 有未保存更改时关闭 sheet，显示“放弃更改”确认。
- 移动使用独立 sheet，展示当前曲目和目标曲库路径，校验必须位于 `library/`、不得包含 `..`；服务端继续执行最终冲突与权限校验。
- 删除使用 `confirmationDialog`，明确曲目名称与不可撤销性，不与编辑器共享成功状态。

## 10. 批量操作协调器

新增独立的 `TrackBatchOperationController`（或同职责模型），不把循环、取消和结果拼在视图中。它发布：

- 操作类型、总数、已处理数和当前曲目。
- 每个曲目的 pending / succeeded / failed / skipped 状态。
- 是否正在执行、是否已请求停止和最终摘要。

服务端没有批量写端点，控制器按稳定的选中顺序一次执行一个请求，保证写负载有界。用户点击“停止”后不再开始下一项；已经进入 Rust/网络层的当前请求允许完成，因为 Swift Task 取消不保证中断 UniFFI 调用。未开始项标记为 skipped，而不是误报失败。

部分失败不中断后续项目，除非用户主动停止。完成或停止后只执行一次专辑刷新；结果面板列出具体曲名、动作和错误，可重试失败项目。刷新失败不能覆盖已经得到的批量结果。操作期间锁定会改变选择集合或再次启动写操作的控件，但播放控制始终可用。

首版通过同一协调器覆盖批量标签更新与批量删除。加入歌单继续使用服务端已有一次多 ID 调用，不伪装成曲库写操作进度。

## 11. 数据流

以管理员批量清空流派为例：

1. `MediaDetailView` 把稳定排序后的选中曲目快照交给批量编辑器。
2. 编辑器产生 `genre = clear` 的字段意图并通过本地校验。
3. `TrackBatchOperationController` 为每首曲目构造 `TagUpdate(clearFields: [.genre])`。
4. core 校验字段冲突并向 `/rest/ext/updateTags?id=…&clear=genre` 发出认证请求。
5. server 的管理员守卫通过后，在事务内把对应覆盖行 upsert 为 `NULL`。
6. 协调器逐项发布进度和结果；播放控制器不参与该数据流。
7. 批量结束后详情模型刷新一次，读取查询根据覆盖行存在且值为 NULL 返回无流派。
8. 曲目表格、搜索与流派集合观察到一致结果。

## 13. 错误与边界状态

- 初始详情失败：完整错误态、实际错误摘要和“重试”。
- 保留内容刷新失败：旧内容继续可播放，顶部非阻塞横幅提供重试。
- 封面 URL 失败：显示稳定占位与局部重试，不清空曲目。
- 编辑校验失败：字段旁显示原因，禁止发送请求。
- 管理请求 401/403：显示权限已变化的明确提示并刷新会话可见状态，不暴露原始响应体。
- 曲目在操作前已被其他管理员删除：标为该曲目失败，继续后续项，最终刷新移除不存在项。
- 操作中切换专辑：请求可以完成，但结果绑定原 album ID；不得覆盖新专辑详情或在新专辑显示旧成功横幅。
- 当前播放曲目被删除：本阶段不强制停止已经装载的本地 AVPlayerItem；后续切曲按现有失败恢复规则处理，刷新后的详情不再显示该曲目。
- 专辑无封面、无流派、无年份、多碟、长标题和未知时长均有稳定呈现。

## 14. 测试策略

### 14.1 contract / server / core

- `TagField` 序列化名称稳定且覆盖允许清空的全部字段。
- core 对保持、设置、清空和混合更新编码正确；拒绝空更新、同字段冲突、清空标题和越界数字。
- 旧式仅设置参数仍保持原行为。
- server 只允许管理员更新；非法 clear 字段、重复冲突和 malformed 数字返回参数错误。
- 数据库区分无覆盖、非空覆盖与 NULL 清空；扫描后清空状态仍保持。
- 清空 genre/year/track/discNumber/album/artist 后，详情、搜索、排序和流派聚合不回退原值、不产生空流派或数字 0。
- 删除覆盖行能恢复文件原标签，证明“恢复来源”与“显式清空”不同。
- `getOpenSubsonicExtensions` 声明同步更新且标准 OpenSubsonic 响应不受影响。

### 14.2 Swift 模型与纯策略

- 初始加载、保留内容刷新和局部封面失败映射到正确 phase。
- 专辑 A 请求晚于 B 返回时，不能覆盖 B 的详情、封面或错误。
- 同一专辑刷新保留内容；切换专辑清除旧内容。
- 单曲字段未变/设置/清空的 patch 正确；标题空白与数字非法时不调用客户端。
- 保存防重复提交，失败保留输入，成功发布一致消息。
- 批量字段的保持/设置/清空产生正确更新；危险字段不出现在批量设置策略。
- 批量部分失败继续、停止后不启动下一项、未开始项标记 skipped、最终只刷新一次。
- 切换专辑后旧批量结果不污染新专辑。
- 成员策略不构造管理动作；管理员动作集合完整。
- 宽/窄列策略、选择保留与空态文案正确。

### 14.3 Swift 视图与真实 macOS 冒烟

使用本地 Garage + Rust 服务端与至少一张多碟专辑验证：

1. 920pt 紧凑详情与宽屏检视器中的头部、表格和批量栏均无裁剪。
2. 快速打开 A/B 两张专辑，B 不闪回 A；刷新失败仍能播放旧内容。
3. 普通成员看不到编辑、封面、移动、删除和可见范围；仍可播放与加入歌单。
4. 管理员设置并清空各类可选标签，刷新、搜索和流派筛选显示一致。
5. 批量更新中观察进度、部分失败详情、停止与失败项重试。
6. 更换封面后无需离开页面即可看到新封面。
7. 移动和删除的独立流程、二次确认及完成后刷新正确。
8. 从专辑中间曲目播放，在详情切换和全部管理流程中声音与全局播放器状态连续。
9. 键盘多选、Command-A、Return/双击播放、Escape 关闭临时界面和 VoiceOver 标签可用。
