# macOS 歌曲拖入歌单设计

- **日期**：2026-07-18
- **范围**：macOS 原生拖放、现有 OpenSubsonic 歌单写接口
- **状态**：已实施

## 1. 目标

当前歌曲只能通过上下文菜单和歌单选择弹窗加入歌单，连续整理专辑或搜索结果时步骤偏多。本切片增加桌面客户端的直接整理路径：从专辑、搜索、收藏或歌单详情拖动一首或多首歌曲，放到侧栏目标歌单即可追加。

## 2. 交互

- 歌曲行可拖动；若当前行属于多选集合，则一次拖动整个选择，否则只拖当前行。
- 多选载荷严格保持当前列表中的视觉顺序，不使用 `Set` 的不稳定顺序。
- 歌单详情按位置选择，必须保留重复曲目的每个实例和原歌单顺序。
- 侧栏只允许歌单叶子接收歌曲；文件夹不接收，也不隐式创建歌单。
- 悬停目标歌单时使用原生强调色背景和加号提示；放下后继续使用现有 `addTracks` 单请求追加并刷新歌单树。
- 目标行显示歌曲数量，成功刷新后提供可见反馈；失败沿用侧栏局部错误，不吞掉错误。
- 写操作进行中拒绝新的放置，避免与歌单重排、删除或其他写操作竞态。

## 3. 数据与边界

- 使用自定义 `Transferable` Codable 载荷，只包含有序的 opaque track ID；不传认证 URL、媒体路径或完整 DTO。
- 自定义 UTType 限定 Yevune 歌曲拖放，避免把 Finder 文本误识别为曲目 ID。
- Swift 只负责平台拖放和交互编排；网络仍调用 `MusicClientProviding.addTracks`，不重复实现 OpenSubsonic 请求。
- 不新增依赖，不改服务端、contract 或 Rust core。

## 4. 边界状态

- 空载荷、空 ID 或全为空的多载荷不接受。
- 来自多个拖动项的载荷按系统交付顺序合并；不去重，因为歌单允许重复歌曲。
- 拖放发起后源列表变化不修改已经编码的快照。
- 目标歌单刷新失败但追加成功时保留服务端成功语义，并展示现有刷新错误；再次刷新可恢复计数。

## 5. 测试与验收

- 纯策略测试：单行、多选顺序、非选中行、重复实例、多个载荷合并、空载荷和写入中拒绝。
- 静态视图测试：四类歌曲列表均提供 `.draggable`，侧栏歌单使用 `.dropDestination`，文件夹不接收。
- 复用 `PlaylistViewModelTests` 验证一次提交全部 ID、单飞与失败反馈。
- 完成前运行 Swift 全量测试、release 构建和启动脚本回归。
- 真实 macOS 窗口已确认歌曲行可发起系统拖动预览，侧栏歌单的数量与可访问性信息正常显示。当前桌面自动化仅提供瞬时坐标拖动，无法稳定模拟系统拖放所需的悬停节奏，因此最终放置与计数刷新由策略测试、静态视图接线测试及既有 `PlaylistViewModelTests` 的单请求/刷新测试共同覆盖；后续人工冒烟测试应补验完整手势。
