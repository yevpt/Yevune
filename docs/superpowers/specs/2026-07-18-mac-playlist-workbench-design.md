# macOS 歌单整理台设计

- **日期**：2026-07-18
- **子项目**：Yevune macOS 原生客户端
- **状态**：依据“现代化客户端、完善曲库管理”的长期方向确认
- **取代范围**：补充并取代 `2026-07-12-mac-multilevel-playlist-design.md` 中“歌单内曲目重排不做”和旧歌单详情界面部分

## 1. 目标

把现有“常驻名称/备注文本框 + 基础曲目 List”升级为适合日常使用的原生歌单整理台：清楚展示歌单身份与统计，提供播放全部、随机播放、显式元数据编辑、曲目多选批量移除，以及保持重复曲目语义的拖动重排。歌单管理状态必须和侧栏、专辑“加入歌单”入口共享，避免同一窗口内出现两份不同步的歌单模型。

## 2. 协议与边界

- 不新增私有协议。完整顺序替换使用 OpenSubsonic `createPlaylist` 的 `playlistId + songId[]` 语义，服务端已有实现会在 SQLite 事务内替换曲目。
- `core` 新增 `update_playlist_metadata` 与 `replace_playlist_tracks` 编排；Swift 不拼 URL、不解析协议响应。
- 重排传递完整的曲目 ID 序列，必须保留重复 ID 和当前顺序。
- 服务端继续强制所有权与曲目外键；客户端不做可绕过的本地持久化。
- 不引入新依赖、数据库或缓存。

## 3. 状态所有权

`LibraryView` 持有窗口内唯一的 `PlaylistViewModel`，同时注入：

- 侧栏歌单树；
- 曲库/专辑详情中的“加入歌单”；
- 歌单详情整理台。

任何成功写操作统一刷新当前详情和树的统计。切换歌单时使用 generation 防止旧详情晚到覆盖新选择；同一写操作 single-flight，操作期间禁止再次改序或移除。

## 4. core API

- `update_playlist_metadata(id, name, comment)`：一次 `updatePlaylist` 同时提交名称与备注。
- `replace_playlist_tracks(id, song_ids) -> PlaylistDetail`：调用 `createPlaylist?playlistId=...&songId=...`，解码服务端返回的完整详情。

名称在客户端与服务端都不得为空白。备注允许空字符串，表示清空备注；不得因 `Option` 合并逻辑意外恢复旧备注。

## 5. 原生界面

### 5.1 头部

- 使用歌单图标、名称、备注、曲目数与总时长组成稳定身份区。
- 主要操作为“播放全部”；次要操作为“随机播放”和“编辑信息”。
- 编辑信息使用 sheet，明确“取消/保存”，名称校验就地显示；不再依赖文本框回车隐式保存。

### 5.2 曲目列表

- 每行显示顺序、标题、艺人、专辑、时长，并保留播放队列右键菜单；单击与 Command 多选完全交给原生 List，避免自定义双击手势抢占系统选择。
- 支持 macOS 多选；选择后在列表底部显示固定批量栏，提供播放所选、移出歌单、清除选择。
- 单曲右键仍可移出，但必须二次确认；批量移除一次提交完整剩余顺序，不能逐条请求造成部分成功。
- 曲目可拖动重排；拖动后的完整顺序一次提交。请求失败回滚到服务端详情并显示局部错误。
- 曲目实例按位置识别，不能以 track ID 作为唯一行身份，否则重复曲目会坍缩。

### 5.3 空与错误状态

- 空歌单显示“从专辑或搜索结果加入歌曲”的行动提示，播放按钮禁用。
- 初始加载、刷新、写入和失败有独立状态；已有内容刷新失败时保留内容。
- 批量栏和全局播放器是垂直兄弟布局，禁止互相覆盖。

## 6. 播放行为

- “播放全部”按歌单当前顺序建立队列。
- “随机播放”先生成新的随机顺序并从第一首开始，不修改歌单持久顺序。
- “播放所选”按歌单中的可见顺序播放，不按选择点击顺序。
- 上一首继续使用全局播放控制器的直接切换语义。

## 7. 测试与验收

- core：元数据同请求编码；完整替换保留重复 `songId` 与顺序；响应详情解码；错误信封。
- Swift ViewModel：旧详情竞态、single-flight、元数据成功/失败、乐观重排成功与失败回滚、批量移除、树与详情同步。
- Swift policy：重复曲目位置身份、move 算法、选择后剩余顺序、播放所选顺序、空白名称校验。
- 真实 macOS：920pt 与宽窗口；编辑/取消；Command 多选；批量栏不与 PlayerBar 冲突；拖动重复曲目；随机播放不修改歌单；成员账户可完成自有歌单整理。
- 完成前运行 Swift 全量测试、三个 Rust crate 的 test/Clippy/fmt、启动脚本与真实界面验收。
