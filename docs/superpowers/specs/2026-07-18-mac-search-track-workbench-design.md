# macOS 搜索曲目整理台设计

- **日期**：2026-07-18
- **子项目**：Yevune macOS 原生客户端
- **状态**：依据“现代化客户端、完善且易操作的曲库管理”长期方向推进

## 1. 问题与目标

搜索是用户定位散落曲目的最快入口，但现有曲目结果只是不可选择的纵向行：只能双击或右键播放，不能直接加入歌单，也不能批量操作。专辑详情虽有歌单选择器，但它是私有视图，写入失败仍会关闭，搜索无法复用。

本切片把搜索结果升级为原生曲目整理台：使用 macOS `List(selection:)` 支持单选、Shift/Command 多选和 Command-A；按搜索结果可见顺序播放所选；一次请求把所选曲目加入目标歌单；单曲右键也能进入相同选择器。选择器由专辑和搜索共享，写入失败保留在原位并显示错误。

## 2. 状态与边界

- `LibraryView` 已持有窗口唯一 `PlaylistViewModel`，继续注入搜索结果和专辑详情，不创建第二份歌单状态。
- 搜索选择以 track ID 保存；搜索模型已经稳定去重，刷新或翻页后仅保留仍可见 ID。
- 播放所选必须按 `model.tracks` 的可见顺序，不按点击顺序。
- 批量加入歌单调用既有标准 OpenSubsonic `updatePlaylist`，把全部 `songIdToAdd` 放在一个请求里；Swift UI 不拼协议 URL。
- 歌单写入 single-flight；成功后刷新树统计，若当前打开同一歌单则刷新详情。
- 不增加依赖、私有服务端端点或本地持久化。

## 3. 原生界面

- 搜索结果整体改为原生 `List`，艺人与专辑继续横向展示，曲目行成为可选择行。
- 曲目行提供显式播放按钮、标题、艺人、专辑、时长；右键保留播放队列动作并增加“加入歌单”。
- 有选择时底部显示固定批量栏：选择数量、播放所选、加入歌单、清除选择。
- 批量栏与全局播放器是不同层级的垂直兄弟，920pt 和宽窗口均不得重叠。
- 共享歌单选择 sheet 展示目标歌单、写入进度和局部错误；空歌单树给出先创建歌单的提示。成功才关闭，取消不写入。
- 成功后搜索页显示非阻塞确认消息并清除选择；失败保留选择以便重试。

## 4. 测试与验收

- policy：选择按可见顺序输出；刷新只保留仍可见 ID；全选使用当前全部结果。
- ViewModel：批量加入一次提交全部 ID；写入 single-flight；失败返回 false 并保留错误；成功刷新树与已打开详情。
- presentation：搜索视图使用共享歌单模型与选择器，专辑不再维护私有副本。
- 真实 macOS：920pt 与宽窗口；单选、Shift/Command 多选、Command-A；批量播放；选择器取消/失败/成功；批量栏与 PlayerBar 共存。
- 完成前运行 Swift 全量测试、三个 Rust crate 的 test/Clippy/fmt、启动脚本与真实界面验收。

## 5. 非目标

- 本切片不增加搜索结果内的标签编辑、删除或访问控制；这些管理员动作继续通过专辑整理台完成。
- 不实现跨窗口拖放、拖到侧栏歌单或智能歌单。
